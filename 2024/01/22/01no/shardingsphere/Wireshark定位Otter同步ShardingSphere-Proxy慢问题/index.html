Wireshark 定位 Otter 同步 ShardingSphere-Proxy 慢问题

问题背景

用户反馈在使用 ShardingSphere-Proxy 进行 Otter 数据同步时，速度较慢，延迟能够到达几个小时。

1.jstack分析线程状态

jstack 分析线程状态正常。

2.async-profiler火焰图分析

在用户环境通过 async-profiler 抓取火焰图，分析执行链路开销。可以发现 com.mysql.cj.jdbc.StatementImpl.setupStreamingTimeout cpu 开销占到 16%。
具体原因是用户在 ShardingSphere-Proxy 的 jdbc url 参数里添加了 netTimeoutForStreamingResults 参数，并且没有开启预编译执行。所以每次 create statement 时 MySQL 驱动都需要下发 netTimeoutForStreamingResults 参数到 mysql server，开销较大。

URL 配置如下:

解决方式：jdbc url 中移除该参数，可以在 mysql server 上进行配置。

  • net_write_timeout 参数
    https://dev.mysql.com/doc/refman/8.0/en/server-system-variables.html#sysvar_net_write_timeout
  • net_read_timeout 参数
    https://dev.mysql.com/doc/refman/8.0/en/server-system-variables.html#sysvar_net_read_timeout

上面参数去除后，用户反馈延迟依然很大，于是继续分析。

3.otter同步慢问题定位

3.1.初步分析是大量执行 set autocommit = 0;delete;commit;操作,理应效率不高.

ShardingSphere-Proxy 里显示 sql 执行时使用的是普通的 statement,没有使用预编译 sql。

Otter 同步的 sql 一条语句对应一个事务，理应比较慢。并且通过抓包确定没有使用 preparedStatement,所以性能更差。

1471 条delete操作，
6228条commit,
6248条 set autocommit=0,
6258条 set autocommit=1

sql执行流程就是重复
Set autocommit=0
Delete
Commit
Set autocommit=1

3.2.Otter 调用 ShardingSphere-Proxy 抓包详细分析

客户环境通过 tcpdump 命令抓包,将包下载下来进行分析。

将3307端口设置成mysql协议端口。
通过 Wireshark 过滤出所有mysql错误包，显示有120条异常包。点击一个异常包里的 error message 再次进行过滤

发现还是120条异常包,说明很可能是这个异常导致的问题。
然后选择一个报错追踪tcp流。

当前frame.number不变,向上找对应的 Request Query 包,可以发现是批量delete


发现上面先批量delete,然后返回npe,基本确定问题.

找到批量delete sql后右键查看tcp flow

看了所有批量delete的语句，全部都执行失败(proxy返回npe,没有具体异常信息，需要加日志)，然后otter rollback事务。
然后otter就会将上面的批量delete语句拆分成一条条delete(每条delete都是单独开启事务执行)，所以性能较差。
下面截图可以看到两次批量delete都返回npe,然后后面就是一条条的delete语句。

可以看到这个tcp flow里批量delete都对应null异常响应,然后会单条sql重试.

可以点击一条异常数据进行查看.

3.3.otter同步逻辑分析

otter报错日志

otter代码分析:
com.alibaba.otter.node.etl.load.loader.db.DbLoadAction#load
大体流程:

otter 会首先一阶段通过 spring template 使用 add batch 方式批量执行 sql 语句，提高数据同步性能。
如果一阶段失败，会单条单条执行 sql 语句，此时性能会变差。

1.ShardingSphere Proxy 没有处理客户端 login request 里设置的 multiple statements 能力标识,导致后续执行批量delete失败.修复并识别即可.
2.后续跑e2e时候发现还是报错.原因是 ShardingSphere Proxy 返回的 Server Greeting 里需要声明 proxy 具有 multiple statements 能力.原因是8.0.33的mysql驱动里会校验

第二点一开始怀疑是和ssl有关系,本地proxy开ssl之后,还是能看到login request user包的,里面传递了 multiple statements,所以和ssl没关系.

和ssl没有关系,proxy没有返回支持 multi statement 的能力,mysql8.0.33驱动之前,没有校验该值,8.0.33里开始校验该值
